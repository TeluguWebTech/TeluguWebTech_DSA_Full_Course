🎯 What is a String?

A String is a sequence of characters. In Java, String is not a primitive data type — it is a class (an object). Examples: "DSA", "Champion". Strings are always enclosed in double quotes.

String name = "Kalyan";   // String literal
String message = "Welcome to DSA!";
🔒 Immutability of Strings

Immutable means cannot be changed after creation. When you modify a String, Java creates a new String object instead of modifying the existing one.

String str = "Kalyan";
str = str + " Suresh";   // A NEW String object is created
// Original "Kalyan" remains unchanged in memory
💡 Important: String objects are immutable — any operation that seems to modify a String actually creates a new String.
📚 Three String Handling Classes

Java provides three classes to handle strings:

  • String – Immutable (cannot change). Best for fixed values.
  • StringBuilder – Mutable, not thread-safe (faster).
  • StringBuffer – Mutable, thread-safe (synchronized, slower).

StringBuilder sb = new StringBuilder("Hello");
sb.append(" World");   // Modifies same object
System.out.println(sb);  // Output: Hello World
🔧 StringBuffer Example (Thread-Safe)
StringBuffer buffer = new StringBuffer("Java");
buffer.append(" Programming");

💾 String Pool & Memory Management

String Pool (String Intern Pool) is a special memory area inside the Heap where Java stores String literals. When you create a String using double quotes, JVM first checks if the same value exists in the pool. If yes, it reuses the reference — saving memory.

📌 Two Ways to Create Strings
  • String Literal: String s1 = "Hello"; → stored in String Pool. Reuses existing objects.
  • Using 'new' keyword: String s2 = new String("Hello"); → stored in Heap memory (outside pool). Always creates a new object.
String a = "Hello";   // Goes to String Pool
String b = "Hello";   // Reuses same object from pool
String c = new String("Hello");  // New object in Heap

System.out.println(a == b);  // true (same reference)
System.out.println(a == c);  // false (different references)
✅ Memory efficiency: String Pool avoids duplicate string objects.
⚖️ Comparing Strings: == vs .equals()
  • == operator compares memory references (addresses).
  • .equals() method compares actual content (character sequence).
String s1 = "Java";
String s2 = "Java";
String s3 = new String("Java");

System.out.println(s1 == s2);      // true (same pool reference)
System.out.println(s1 == s3);      // false (different memory)
System.out.println(s1.equals(s3)); // true (same content)
📌 Always use .equals() for content comparison in real applications.
🎤 Interview Questions & Answers

❓ Q1: Why are Strings immutable in Java?

Answer: Strings are immutable for security, caching (String Pool), thread-safety, and performance. Immutability allows String Pool to work efficiently — JVM can cache String literals safely. It also makes Strings inherently thread-safe and prevents accidental modification.

❓ Q2: What is the difference between StringBuilder and StringBuffer?

Answer: Both are mutable, but StringBuffer is thread-safe (synchronized methods) — suitable for multi-threaded environments. StringBuilder is not thread-safe but faster, ideal for single-threaded operations. Use StringBuilder for most scenarios unless thread-safety is required.

❓ Q3: What is the String Pool? Where is it located?

Answer: String Pool (String Intern Pool) is a special area inside the Heap memory where Java stores string literals. When a new literal is created, JVM checks the pool first. If found, reference is reused; otherwise, a new object is added to the pool. This optimizes memory usage.

❓ Q4: What is the difference between creating a String with new vs literal?

Answer: String s = "text"; → stored in String Pool, may reuse existing object. String s = new String("text"); → always creates a new object in Heap memory (outside pool), even if same value exists. Literal is more memory-efficient.

❓ Q5: Why should we use .equals() instead of == for string comparison?

Answer: == compares memory references, not content. .equals() compares actual character sequence. For user input or dynamic strings, references may differ even if content is same, so .equals() gives correct result.

❓ Q6: What happens when you concatenate many Strings using + in a loop?

Answer: Each concatenation creates a new String object (due to immutability), leading to O(n²) time and memory overhead. For multiple concatenations, use StringBuilder or StringBuffer for efficiency.

// Bad: creates many intermediate String objects
String result = "";
for (int i = 0; i < 1000; i++) {
    result += i;   // DON'T DO THIS
}

// Good: use StringBuilder
StringBuilder sb = new StringBuilder();
for (int i = 0; i < 1000; i++) {
    sb.append(i);
}
String result = sb.toString();

📌 Summary & Key Takeaways
  • ✔️ String is immutable — stored in String Pool (literals) or Heap (new keyword).
  • ✔️ StringBuilder → mutable, fast, not thread-safe (single-threaded).
  • ✔️ StringBuffer → mutable, thread-safe, slower (multi-threaded).
  • ✔️ Use .equals() for content comparison, == for reference comparison.
  • ✔️ String Pool saves memory by reusing identical literals.
  • ✔️ For heavy concatenation, always use StringBuilder/StringBuffer.

🎯 Interview Tip: Be ready to explain immutability benefits (security, caching, thread-safety). Also remember that String literals are interned automatically, but you can call .intern() manually on heap strings to move them to pool.
